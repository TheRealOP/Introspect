AI Provider Resolution — src/server/ai/provider.ts new
Design goal: anyone with any AI subscription can use Introspect. Self-hosters set their own env vars. Hosted users pay for ours. A DB settings row overrides env — so a user who pastes their own key in the UI gets their model, everyone else falls through to the server's env default (currently Groq). No change to any AI behavior — just where the model comes from.
Resolution order — getAiConfig(db)
1
DB settings row (introspect_settings singleton, id = "default")
If provider, model, apiKey are all set → use these. Allows in-app key entry.
2
Env variables — AI_PROVIDER / AI_MODEL / per-provider key
Fallback when no DB row exists. Used by hosted version and developers.
3
Hard default — Groq + llama-3.3-70b-versatile
Preserves existing Phase 3 behavior if nothing else is configured.

AI Profile — 5 Buckets new — src/server/ai/insights.ts
Built by buildInsights(db, habits, recentEntries, nudgeStats). Uses same tool-calling pattern as extract.ts. Cached in introspect_profile singleton — not regenerated on every page load. User manually triggers refresh; UI shows a "stale" indicator when new entries arrived since last build.
🟢 Aware
Habits the user explicitly names in their check-ins. They know about these.
🟡 Unaware
Recurring patterns across entries the user never called out by name. Hidden habits the AI surfaces.
🔵 Want to Build
Inferred from positive sentiment + stated intent in entries. Things they're trying to do more of.
🔴 Want to Remove
Inferred from negative sentiment + self-criticism. Habits they want gone.
🟣 Suggestions
Small complementary habits the AI proposes based on what it sees working. Not from the data — forward-looking.
+ nudgePreference
A short note on what kinds of nudges this user actually picks — derived from their selected=1 nudge history. e.g. "Prefers sensory-reset nudges; rarely picks pattern-tied ones." Feeds back into future extractFromEntry calls.
