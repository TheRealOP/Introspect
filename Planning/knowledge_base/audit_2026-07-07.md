# Full Project Audit — 2026-07-07

A complete audit of the Introspect codebase performed on the `develop` branch (HEAD `ba3f300`). Every server file, API route, tRPC router, the AI layer, auth, middleware, DB schema, and service worker were reviewed; `tsc --noEmit` passes cleanly. Findings are ordered by severity. Status boxes should be checked off as items are fixed.

---

## Overall Assessment

The codebase is structurally healthy: clean separation between routers / AI layer / DB layer, consistent style, sensible patterns (structured-output fallback chain in `src/server/ai/structured.ts`, honeypot on feedback, graceful email fallbacks). However, at audit time there were **two bugs that break core features in production, one privacy incident already committed to GitHub, and one credential-exposure issue**.

---

## 🔴 Critical Findings

### C1. New signups get a broken database (fix exists, unmerged)

- [ ] Fixed

`schema.ts` defines `mode` and `tier` columns on `introspect_settings`, but `USER_DB_INIT_SQL` in `src/app/api/signup/route.ts` creates the table **without** them. Drizzle always selects every mapped column, so for any user who signed up after those columns were added, `settings.get`, `getAiConfig`, and therefore **journal analyze, chat, and insights all fail** with `no such column: mode`.

**Fix already written** on the unmerged branch `worktree-fix+settings-schema-columns` (commit `f60d4fe`, worktree at `.claude/worktrees/fix+settings-schema-columns`). It adds the columns to the init SQL and runs lazy `ALTER TABLE` migrations in `createTRPCContext`. It predates the `ensureUserTables` call now in `trpc.ts`, so it needs a small rebase before merging.

### C2. Reminders cron endpoint is dead in production (middleware blocks it)

- [ ] Fixed

`src/middleware.ts` only whitelists `/auth`, `/api/auth`, `/api/signup`, `/api/verify`. `/api/cron/reminders` is not public, so an external cron caller (no session cookie) gets a 302 redirect to the sign-in page — the handler never runs, and the caller sees a "successful" 200 from sign-in HTML, so it fails silently. The `CRON_SECRET` guard inside the route is never reached.

Same matcher also breaks:
- `/api/feedback` — the route supports anonymous feedback (optional session/email) but middleware blocks all logged-out submissions.
- `/manifest.webmanifest`, `/sw.js`, icon PNGs — manifest fetches are made without cookies by default, so PWA install metadata redirects to sign-in.

Also: **no `vercel.json` exists**, so Vercel Cron is not configured at all — reminders currently have no trigger in production.

### C3. Real user database committed to the repo

- [ ] Fixed (removed + history purged + secrets rotated)

`user-704da976-1bf9-4d61-9475-b238138222df.sqlite` (94 KB — journal entries, habits, chat history, possibly an AI API key in its settings table) was committed in `935cfe3` ("yayay commit probably") and is still tracked. If the repo is public, this is a live privacy leak.

Remediation:
1. `git rm --cached user-*.sqlite`
2. Add `user-*.sqlite` and `.claude/` to `.gitignore` (the `.claude/worktrees/` dir is also untracked and at risk of being committed)
3. Purge from history (`git filter-repo` or BFG)
4. Rotate anything sensitive that was inside it

## 🟠 High Findings

### H1. Never-expiring Turso auth token shipped to the browser

- [ ] Fixed

`src/server/auth.ts` copies `dbUrl` and `dbAuthToken` into the session object, and `layout.tsx` wraps the app in `SessionProvider` — so `/api/auth/session` hands the client a **permanent** (`expiration=never`, see `src/server/turso.ts`) full-access token to the user's database. Any XSS, malicious extension, or shared computer becomes irrevocable DB access.

Fix: keep both fields in the JWT only (server-readable); strip them in the `session` callback. Longer term, provision tokens with an expiry instead of `never`.

### H2. Signup has no abuse protection; provisions paid infra before verification

- [ ] Fixed

- Anyone can hammer `/api/signup` and create unlimited Turso databases (each with a never-expiring token) without verifying an email.
- No rate limiting anywhere (signup, signin, resend-verification, feedback).
- The 409 "Email already registered" response is an email-enumeration oracle — inconsistent with `/api/verify/resend`, which deliberately never leaks registration status.

Fix: rate-limit by IP; consider deferring DB provisioning until first verified sign-in.

## 🟡 Medium Findings

### M1. Table DDL defined in three places — drift already happened

- [ ] Fixed

`schema.ts` (Drizzle), `USER_DB_INIT_SQL` (signup route), and `ensure-tables.ts` each hand-maintain CREATE TABLE statements. Finding C1 is the direct result. Consolidate: generate init SQL from the Drizzle schema, or keep one shared migration list that both signup and the lazy-migration path run.

### M2. Wiki updates after chat likely lost on Vercel

- [ ] Fixed

`src/app/api/chat/route.ts` fires `extractWikiUpdates(...)` as a floating promise inside `onFinish`. On serverless, the function can be frozen as soon as the response stream closes, silently killing the background wiki write. Use `waitUntil()` from `@vercel/functions` or await before the stream resolves.

### M3. Legacy user DBs only partially healed

- [ ] Fixed

`ensureUserTables` creates only push/reminders tables, and the chat route never calls it — user DBs created before chat/wiki existed will 500 in chat (`chat_messages`, `wiki_pages`, `wiki_edges` missing). The `SETTINGS_MIGRATIONS` pattern on the fix branch is the right home for all lazy migrations. Also: `ensure-tables.ts` falls back to cache key `"unknown"` — if neither `config.url` nor `filename` resolves, subsequent DBs are skipped entirely.

### M4. `CRON_SECRET` accepted as a query parameter

- [ ] Fixed

`src/app/api/cron/reminders/route.ts` accepts `?secret=` — URLs land in access logs and proxies. Keep only the `Authorization: Bearer` header path.

### M5. `analyzeAll` is an unbounded serial AI loop

- [ ] Fixed

`journal.analyzeAll` makes one AI call per un-analyzed entry inside a single serverless invocation — will hit function timeout with a moderate backlog. Cap batch size or chunk invocations.

### M6. `testConnection` / custom provider is a mild SSRF vector

- [ ] Acknowledged / documented as intentional

An authenticated user can make the server POST to any `baseUrl`. Acceptable for a BYO-provider product; consider blocking private/link-local IP ranges if the app ever seriously multi-tenants.

## 🟢 Low / Hygiene

- [ ] **Default AI provider is `ollama` at `localhost:11434`** (`provider.ts`) — a production user with no settings row and no `AI_PROVIDER` env silently points at a nonexistent localhost server; fails with a confusing error.
- [ ] **`JSON.parse` of `tags`/`habitTags` without try/catch** in `wiki.ts`, `chat/route.ts`, `insights.ts` — one malformed row bricks the whole query.
- [ ] **Everything is `publicProcedure`** while the tRPC context itself throws UNAUTHORIZED — works, but misleading; add a `protectedProcedure` alias. Also, expired sessions hitting `/api/trpc` get a 302 HTML redirect from middleware instead of a clean 401.
- [ ] **Verification tokens stored in plaintext** in the users DB — hash them to defend against DB-read attackers (minor at current scale).
- [ ] **No tests, no CI, no lint script** — `package.json` has `typecheck` only. A GitHub Action running `tsc --noEmit` + `next build` on PRs would catch schema/type drift.
- **`createUserDb` cache keyed by URL only** — fine today (URL is per-user); don't reuse URLs with different tokens.
