Flow 1 · Entry Analysis — triggered by journal.analyze

Runs once per check-in, after the user submits text. Extracts behavioral habits and generates 4 micro-action nudges.

① Trigger
User submits check-in text in journal.tsx
journal.create inserts the entry row
journal.analyze({ entryId }) mutation fired immediately after
→
② Data gathering
Fetch this entry content from DB
Fetch last 5 previous entries (ordered ascending → context window)
Fetch last 10 selected nudges (user's revealed preferences)
Call resolveAi(db) → { model, mode }
→
③ extractFromEntry() — src/server/ai/extract.ts
Delegates to generateStructured() (Vercel AI SDK v6)
Mode comes from settings: auto, tool, or json
auto tries forced tool call, then JSON mode, then text+JSON parse
Temperature: 0.2 (very deterministic)

System prompt: behavioral coach, strict nudge rules (4 types: physical, close-the-loop, sensory reset, tiny progress)
User prompt: previous entries (oldest-first) + past selected nudges + latest entry

Output schema (Zod):
 · habits[] → name (≤3 words, capitalized) + sentiment
 · nudges[4] → exactly 4 strings, max 20 words each
→
④ DB writes
For each habit: upsertHabit()
 · Case-insensitive name match (lower(name) = ?)
 · If exists: increment occurrences, update lastSeen
 · If new: insert row with occurrences=1
 · Always: insert habit_occurrences row (for streak tracking) using entry.createdAt as seenAt

For nudges: insert 4 rows into introspect_nudges
 · All start with selected=0
 · Linked to this entry via entryId
Why tool-calling instead of JSON mode?
Some providers/models reject strict JSON schema or tool calls. src/server/ai/structured.ts centralizes that variability: mode tool forces a Zod-backed tool call, mode json uses generateObject, and mode auto falls back from tool → JSON → text+parse. Routers stay provider-agnostic.
Flow 2 · Profile Generation — triggered by insights.refresh

Builds (or rebuilds) a single cached "knowledge-graph profile" of the user. Runs on demand — user clicks Refresh on the Insights page. The profile is a singleton row (id = "default") in introspect_profile.

① Trigger
User visits Insights page
UI checks insights.get() — if isStale: true or no profile, a Refresh button appears
User clicks → insights.refresh mutation
→
② Data gathering
Fetch all habits from DB
Fetch all habit_occurrences → compute streaks in JS (computeStreaks())
Streaks: dedupe by calendar day (UTC), find consecutive days, check if active (today or yesterday)
Fetch last 10 entries (reversed → oldest-first for context)
Fetch top 10 most-selected nudges with pick counts
Fetch current total entry count (for staleness stamp)
→
③ buildInsights() — src/server/ai/insights.ts
Same generateStructured() path as extraction
Temperature: 0.2

System prompt: behavioral scientist, tag rules
User prompt: habits block (name, sentiment, occurrences, current streak, longest streak) + recent check-ins + top picked nudges with counts

Output schema (Zod):
 · summary — 3-sentence brief: top positive, top negative, trajectory
 · habitTags[] — each habit gets 1+ tags:
   aware user names it explicitly
   unaware pattern visible but unnamed
   wantToBuild positive / wants more
   wantToRemove negative / wants to stop
 · suggestions[] — 2–4 NEW habit ideas (not in list)
 · nudgePreference — one sentence on nudge style
→
④ Upsert singleton profile
Check if profile row id="default" exists
If yes → UPDATE; if no → INSERT

Arrays stored as JSON strings:
 · habitTags → JSON.stringify([{name, tags[]}])
 · suggestions → JSON.stringify(string[])

Stamps entryCount = current total entries
Stamps updatedAt = unix timestamp now
Staleness Detection
How isStale works
insights.get() reads the profile row and compares
profile.entryCount (stamped at last refresh) vs
COUNT(*) from introspect_entries (live count)

If they differ → isStale: true
If no profile row exists + entries > 0 → also isStale: true
UI surfaces a "Refresh" chip to prompt the user
Reading the profile
JSON columns are parsed back on read:
JSON.parse(row.habitTags) → {name, tags[]}[]
JSON.parse(row.suggestions) → string[]

nudgePreference and summary are plain text — no parsing needed
Streak Algorithm — computeStreaks() in insights.ts
Input: array of unix-second timestamps from habit_occurrences.seenAt
1. Filter nulls, convert to YYYY-MM-DD strings (UTC)
2. Deduplicate (multiple sightings same day = 1 day), sort ascending
3. Walk the sorted days: if curr - prev == 1 day, extend run; else reset run to 1
4. Track longest = max run seen
5. Current streak: if the last recorded day is today or yesterday → return current run; else 0
Why yesterday? A user who logged yesterday but not yet today shouldn't lose their streak mid-day.
